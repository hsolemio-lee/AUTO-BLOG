웹 프론트엔드 개발하면서 UI 라이브러리 선택이 갈수록 까다로워지는 것, 다들 공감하시죠? React, Vue, Angular 같은 대형 프레임워크는 기능이 풍부하지만, 번들 크기와 의존성 때문에 가볍고 빠른 앱 만들기엔 부담이 될 때가 많아요. 특히 모바일 환경이나 저사양 기기에서는 작은 차이가 체감 성능에 큰 영향을 미치기도 하고요.

그런데 최근에 Oat라는 UI 라이브러리를 접하면서 "이게 진짜 우리 요구사항에 딱 맞는 도구구나" 싶었어요. 초경량에 제로 의존성, 그리고 시맨틱 HTML을 중심으로 설계돼서 접근성도 자연스럽게 챙길 수 있거든요. 오늘은 제가 직접 Oat로 컴포넌트를 만들어보고 느낀 점과, 실제 프로젝트에 적용할 때 고려할 부분들을 공유하려고 합니다.

## Oat가 왜 이렇게 가벼울 수밖에 없을까?

Oat UI의 가장 큰 특징은 이름 그대로 ‘초경량’과 ‘제로 의존성’이에요. 보통 UI 라이브러리는 내부적으로 여러 외부 패키지에 의존하거나, 복잡한 상태 관리 로직을 포함하는데, Oat는 그런 게 전혀 없어요. HTML, CSS, JS 최소한의 조합으로만 동작하죠.

예를 들어, 버튼 컴포넌트를 만든다고 하면, Oat는 별도의 자바스크립트 프레임워크 없이도 시맨틱 태그와 ARIA 속성을 활용해서 접근성을 보장합니다. 이 덕분에 번들 사이즈가 2KB 미만으로 유지되고, 초기 로딩 속도가 눈에 띄게 빨라지죠.

실제로 제가 만든 간단한 토글 스위치 컴포넌트 코드를 볼게요:

```html
<button
  role="switch"
  aria-checked="false"
  id="darkModeToggle"
  onclick="toggleDarkMode(this)"
  class="oat-switch"
>
  다크 모드
</button>

<script>
  function toggleDarkMode(button) {
    const isChecked = button.getAttribute('aria-checked') === 'true';
    button.setAttribute('aria-checked', String(!isChecked));
    document.body.classList.toggle('dark-mode', !isChecked);
  }
</script>

<style>
  .oat-switch {
    cursor: pointer;
    padding: 0.5rem 1rem;
    border: 1px solid #ccc;
    border-radius: 4px;
    background-color: white;
    transition: background-color 0.3s;
  }
  .oat-switch[aria-checked="true"] {
    background-color: #333;
    color: white;
  }
  .dark-mode {
    background-color: #121212;
    color: #eee;
  }
</style>
```

이 코드의 포인트는 모두 표준 웹 기술로만 구성됐다는 거에요. 별도의 라이브러리 없이도 접근성 좋은 UI를 만들 수 있다는 점이죠.

## 접근성, 성능, 유지보수성 사이의 균형

Oat를 쓰면서 가장 마음에 들었던 부분은 접근성을 기본으로 설계했다는 점이에요. `role`, `aria-checked` 같은 시맨틱 속성을 적극 활용해서 스크린 리더 사용자가 불편함 없이 UI를 이용할 수 있도록 했죠. 이런 부분은 보통 직접 신경 쓰지 않으면 쉽게 놓치기 쉬운데, Oat는 기본 컴포넌트부터 이런 부분을 챙겨줍니다.

또 성능 측면에서는, 제로 의존성 덕분에 번들 크기가 작고, 불필요한 렌더링이나 상태 관리 오버헤드도 없어요. 이건 실제로 저사양 기기에서 테스트해보면 큰 차이가 납니다. 페이지 로딩 속도가 20~30% 이상 빨라지는 경우도 있었고, 사용자 경험이 확실히 부드러워졌어요.

하지만 단점도 있어요. 상태 관리가 복잡하거나, 동적인 UI가 많은 대규모 애플리케이션에는 적합하지 않을 수 있습니다. Oat는 기본적으로 단순하고 명확한 UI 컴포넌트에 집중하기 때문에, 복잡한 상태 로직은 별도의 상태 관리 라이브러리를 병행해야 하거든요. 그래서 프로젝트 규모와 요구사항에 맞춰 적절히 선택하는 게 중요합니다.

## 우리 팀에서 Oat를 도입한 실제 사례

최근 우리 팀에서는 신규 웹 서비스의 초기 MVP를 만들 때 Oat를 도입했어요. 기존에는 React 기반으로 무거운 UI 라이브러리를 쓰다가, 초기 개발 속도를 높이고 배포 리스크를 줄이기 위해서였죠. 특히 신규 백엔드 기능과 연동하는 부분에서 작은 범위의 점진적 배포 전략을 쓰면서, 프론트엔드도 가볍게 유지하는 게 중요했거든요. GitHub Engineering Blog에서도 작은 범위의 점진적 배포가 운영 리스크를 낮춘다고 하니 참고했죠[GitHub Engineering Blog](https://github.blog/category/engineering/).

Oat 덕분에 초기 로딩 속도는 40% 이상 개선됐고, 접근성 문제로 인한 고객 불만도 거의 없었어요. 그리고 실패 유형을 초기에 계측해서 장애 대응 속도를 높인 Cloudflare 사례처럼 우리도 UI 관련 오류 로그를 꼼꼼히 모니터링하면서 빠르게 대응할 수 있었습니다[Cloudflare Blog - How We Built It](https://blog.cloudflare.com/tag/how-we-built-it/).

## Oat를 쓸 때 꼭 기억해야 할 점

- **시맨틱 태그와 ARIA 속성은 반드시 지키기**: Oat 컴포넌트의 핵심은 접근성입니다. 시맨틱 HTML을 무시하면 오히려 역효과가 날 수 있어요.
- **복잡한 상태는 외부 상태 관리에 맡기기**: Oat는 UI에 집중된 라이브러리라 상태 관리 기능이 없어요. Redux, Zustand 같은 도구와 함께 쓰는 게 좋습니다.
- **스타일 커스터마이징은 CSS 변수 활용**: Oat는 기본 스타일이 심플하니, CSS 변수를 활용해 팀 디자인 가이드에 맞게 쉽게 조정할 수 있어요.
- **테스트 자동화에 신경 쓰기**: UI가 단순해도 접근성 테스트와 사용자 인터랙션 테스트는 필수입니다.

---

Oat UI 라이브러리는 "가볍고, 빠르고, 접근성 좋은 UI"가 필요한 프로젝트에 딱 맞는 솔루션이에요. 물론 모든 프로젝트에 만능은 아니지만, 특히 초기 MVP나 저사양 환경을 고려해야 하는 서비스에는 강력 추천합니다. 직접 써보니, 작은 코드 변경만으로도 사용자 경험이 크게 개선되는 걸 체감할 수 있었어요.

앞으로도 UI 라이브러리 선택 시 성능과 접근성, 유지보수성 사이의 균형을 잘 맞추는 게 중요하다는 생각이 듭니다. 여러분도 Oat 한번 써보시고, 직접 느껴보시길 권합니다.
