### "아니, LocalStack 커뮤니티 에디션이 사라진다고?"

얼마 전 우리 팀에서 AWS 로컬 테스트 환경으로 애용하던 LocalStack 커뮤니티 에디션이 공식적으로 중단된다는 소식을 접했을 때 솔직히 당황스러웠다. 특히 인프라 비용이나 실제 AWS 리소스 사용 없이도 로컬에서 빠르게 개발하고 검증하던 워크플로우가 완전히 바뀌게 생겼으니까.

이 경험을 공유하는 이유는, 나처럼 LocalStack에 의존해 온 개발자들이 앞으로 어떻게 대응할지, 그리고 실제 운영 환경에 미치는 영향을 현실적으로 고민해보는 계기가 되었기 때문이다.

---

## LocalStack 커뮤니티 에디션 중단, 왜 이렇게 큰 파장이었을까?

LocalStack은 AWS 서비스를 로컬에서 흉내 내는 도구로, Lambda, S3, DynamoDB 등 주요 AWS API를 거의 그대로 사용할 수 있다. 덕분에 CI/CD 파이프라인이나 개발 단계에서 실제 AWS에 배포하지 않고도 테스트를 돌릴 수 있다는 점이 큰 장점이었다.

하지만 이번에 LocalStack 공식이 커뮤니티 에디션을 중단하고 유료 버전으로 전환하면서, 무료로 쓸 수 있던 기능들이 제한되거나 사라졌다. 이게 왜 문제냐면:

- **비용 부담 증가**: 작은 스타트업이나 개인 개발자 입장에선 비용이 부담될 수밖에 없다.
- **테스트 환경 재구성 필요**: 기존에 LocalStack에 맞춰진 테스트 스크립트나 CI 설정을 다시 맞춰야 한다.
- **개발 속도 저하 우려**: 실제 AWS 리소스 사용 없이 빠르게 돌리던 테스트가 느려지거나 더 복잡해질 수 있다.

우리 팀도 LocalStack 없이는 Lambda 함수나 S3 연동 테스트가 불편해져, 당장 개발 효율이 떨어질까 봐 걱정이 컸다.

## LocalStack 없이 AWS 서비스 테스트, 현실적인 대안은?

LocalStack 커뮤니티 에디션이 사라진다고 해서 AWS 서비스 테스트를 포기할 수는 없다. 그래서 몇 가지 대안을 찾아봤다.

### 1. AWS SAM CLI의 로컬 기능 활용하기

AWS SAM(Serverless Application Model)은 Lambda 함수와 API Gateway를 로컬에서 실행할 수 있게 해준다. 특히 `sam local invoke`나 `sam local start-api` 명령어로 로컬 테스트가 가능하다.

```bash
sam local invoke MyFunction --event event.json
```

이 방법은 Lambda 함수 단위 테스트에 적합하고, Docker 컨테이너 기반이라 실제 환경과 비슷한 실행 환경을 제공한다. 다만 S3, DynamoDB 같은 다른 서비스는 별도로 모킹하거나 실제 AWS 리소스를 써야 한다는 단점이 있다.

### 2. AWS 공식 SDK 모킹 라이브러리 사용

AWS SDK를 사용하는 코드라면 `aws-sdk-client-mock` 같은 라이브러리로 API 호출을 모킹할 수 있다. 이 방법은 테스트 속도가 빠르고 CI 환경에 부담이 적지만, 실제 AWS 서비스 동작과 완벽히 같지는 않다.

### 3. 실제 AWS 계정 내 테스트 리소스 활용

가장 확실한 방법은 실제 AWS 계정에 테스트용 리소스를 만들어 사용하는 것이다. 다만 비용 관리와 리소스 정리가 필요하고, 테스트 실패 시 실제 비용이 발생할 수 있어 주의해야 한다.

이때 AWS Well-Architected Framework에서 권장하는 "작은 범위의 점진적 배포 전략"을 적용하면, 신규 기능을 소규모로 배포해 운영 리스크를 줄일 수 있다[참고](https://docs.aws.amazon.com/wellarchitected/latest/framework/welcome.html).

## LocalStack 중단이 우리 운영에 미치는 영향과 대응

LocalStack 덕분에 개발 중에 발견하던 여러 오류가 실제 AWS 배포 전에 잡혔다. 이제 이 기능이 사라지면 다음과 같은 변화가 예상된다.

- **장애 대응 속도 저하 우려**: 실패 유형을 사전에 계측하지 못하면 장애 원인 분석이 늦어진다. AWS Architecture Blog에서 강조하는 것처럼, 초기부터 실패 유형을 계측하는 설계가 유지보수성과 장애 대응 속도를 높인다[참고](https://aws.amazon.com/blogs/architecture/).
- **CI/CD 파이프라인 복잡도 증가**: 로컬 테스트가 줄어들면 실제 AWS 환경에서 테스트하는 빈도가 늘어나 배포 시간이 길어질 수 있다.
- **비용과 리소스 관리 부담**: 테스트용 AWS 리소스가 많아질수록 비용과 관리 부담이 커진다.

우리 팀은 이를 극복하기 위해 다음과 같은 실험을 진행했다.

- 테스트용 AWS 계정을 별도로 분리해 비용과 리소스 관리를 명확히 했다.
- 실패 케이스별로 CloudWatch Logs와 X-Ray를 적극 활용해 장애 원인 추적을 강화했다.
- 중요한 기능은 AWS SAM CLI로 로컬 테스트를 병행해 빠른 피드백을 유지했다.

## LocalStack 커뮤니티 에디션 대체 도구, 혹은 직접 구축할 수 있을까?

LocalStack의 빈자리를 메우려는 시도도 많다. 예를 들어 `Moto`는 Python 기반 AWS API 모킹 라이브러리로, 일부 서비스만 지원하지만 간단한 테스트에 유용하다.

직접 AWS API를 흉내 내는 도구를 만드는 것도 가능하지만, AWS API 스펙이 방대하고 자주 바뀌기 때문에 유지보수가 어렵다.

그래서 결국 유료 LocalStack 버전을 도입하거나, 위에서 언급한 AWS 공식 도구와 실제 AWS 리소스를 조합하는 방향이 현실적이다.

## LocalStack 중단을 겪으며 정리한 내 경험적 조언

1. **테스트 커버리지를 다시 점검하라**: LocalStack에 의존했던 부분이 실제 AWS 환경에서 어떻게 테스트되는지 확인해야 한다.
2. **작은 범위부터 배포해 리스크를 줄여라**: AWS Well-Architected Framework에서 제안하는 점진적 배포 전략을 적극 활용하자.
3. **장애 로그와 계측에 투자하라**: 실패 유형을 초기에 잘 계측하면 장애 대응이 훨씬 수월해진다.
4. **도구 의존도를 분산하라**: 하나의 도구에 너무 의존하지 말고, 여러 테스트 전략을 병행하는 게 중요하다.

---

LocalStack 커뮤니티 에디션 중단은 분명 불편한 변화지만, 오히려 AWS 서비스 테스트의 본질적인 문제를 다시 생각해보는 기회가 됐다. 나처럼 LocalStack에 의존해 온 동료 개발자라면, 이번 기회에 테스트 환경과 운영 전략을 한 번 점검해보길 권한다.
