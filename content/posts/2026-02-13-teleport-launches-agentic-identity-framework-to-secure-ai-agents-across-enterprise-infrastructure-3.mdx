### AI 에이전트가 기업 인프라에서 내 마음대로 움직인다면?

요즘 AI 코딩 에이전트가 반복적인 보일러플레이트 작업을 척척 해내면서 개발 생산성이 확실히 올라가고 있죠. 하지만 이들이 기업 내부 시스템에 접속해 자동으로 작업을 수행할 때, 누가 어떤 권한으로 무슨 작업을 했는지 추적하는 게 만만치 않습니다. 특히 AI가 복잡한 아키텍처 설계까지 완전히 대신하기는 아직 어렵고, 사람이 판단해야 할 부분이 많으니 보안 사고가 터지면 치명적일 수밖에 없어요.

Teleport가 최근 발표한 Agentic Identity Framework는 이런 문제를 해결하기 위해 AI 에이전트의 신원을 명확히 하고, 권한을 세밀하게 통제하는 새로운 보안 모델입니다. 이번 글에서는 이 프레임워크를 실제 백엔드 환경에 적용하는 과정을 동료 엔지니어에게 설명하듯 자세히 풀어보겠습니다.

---

## Teleport Agentic Identity Framework가 기존 인증과 다른 점

기존에는 AI 에이전트가 기업 인프라에 접근할 때 일반 사용자 계정이나 서비스 계정을 공유해서 썼는데, 이러면 누가 어떤 AI 에이전트인지 구분하기 어렵고 권한 남용 위험도 큽니다. Teleport의 Agentic Identity Framework는 AI 에이전트마다 고유한 신원(identity)을 부여하고, 각 신원에 맞는 권한 정책을 적용할 수 있게 설계됐어요.

이 프레임워크는 다음 세 가지 핵심 요소로 구성됩니다:

- **에이전트 신원 발급**: AI 에이전트별로 고유한 ID와 인증서 발급
- **권한 정책 관리**: 역할 기반 접근 제어(RBAC) 혹은 속성 기반 접근 제어(ABAC) 정책 적용
- **행위 추적 및 감사**: 에이전트 활동 로그를 상세히 기록해 보안 사고 대응 가능

현실적으로, AI 에이전트가 여러 서버, DB, 클라우드 서비스에 동시다발적으로 접근할 때 이 체계가 없으면 누가 뭘 했는지 전혀 알 수 없고, 권한도 과도하게 부여되는 경우가 많습니다.

---

## 엔터프라이즈 환경에서 Agentic Identity Framework 적용하기

우리 팀에서 Teleport Agentic Identity Framework를 도입하면서 겪은 경험을 공유할게요. 우선, AI 에이전트마다 별도의 인증서를 발급하는 과정부터 시작했습니다. 예를 들어, AI 코딩 에이전트가 자동으로 배포 스크립트를 실행하려면 `deploy-agent`라는 신원을 만들고, 이 신원에만 배포 권한을 줍니다.

```bash
# Teleport CLI로 AI 에이전트 신원 생성 예시
teleport users add deploy-agent --roles=deploy

# 인증서 발급
teleport auth sign --user=deploy-agent --out=deploy-agent-cert.pem
```

이렇게 발급된 인증서는 AI 에이전트가 API 호출 시 TLS 클라이언트 인증에 사용돼, 실제로 누가 요청했는지 서버가 명확히 알 수 있습니다. 이 과정에서 중요한 건 인증서 만료 기간을 너무 길게 잡지 않는 겁니다. 우리 팀은 24시간으로 제한했는데, 너무 길면 탈취 시 위험하고 너무 짧으면 잦은 갱신이 번거롭더라고요.

권한 정책은 Teleport의 RBAC 기능을 활용해 세밀하게 쪼갰습니다. 예를 들어, `deploy-agent`는 `k8s:deploy` 권한만, `monitor-agent`는 로그 조회 권한만 갖도록 했죠. 이렇게 권한을 최소 권한 원칙에 맞게 설계하는 게 보안 사고 예방의 핵심입니다.

## AI 에이전트가 생성한 코드, 보안 점검은 필수

AI 코딩 에이전트가 생산성을 올려주는 건 분명하지만, 생성된 코드가 무조건 안전한 건 아닙니다. 실제로 AI가 만든 코드는 동작 여부와 별개로 보안 취약점, 의존성 문제, 테스트 커버리지 부족 등 여러 리스크가 존재합니다. [TypeScript 5.x Release Notes](https://devblogs.microsoft.com/typescript/announcing-typescript-5-4/)에서도 지적하듯, AI 생성 코드는 별도의 검증 프로세스가 반드시 필요해요.

우리 팀은 Teleport Agentic Identity Framework와 함께, AI가 생성한 코드를 자동으로 스캔하고, 취약점이 발견되면 해당 에이전트 신원을 차단하는 워크플로우를 구축했습니다. 예를 들어, CI 파이프라인에서 `npm audit`과 정적 분석 도구를 돌려서 문제가 있으면 배포를 막고, Teleport API를 통해 해당 에이전트의 인증서를 즉시 폐기합니다.

```bash
# 취약점 발견 시 AI 에이전트 인증서 폐기 예시
teleport auth revoke --user=deploy-agent
```

이런 자동화 덕분에 AI 에이전트가 무분별하게 권한을 남용하는 걸 막을 수 있었고, 보안 사고 발생률이 눈에 띄게 줄었어요.

## Agentic Identity Framework 운영하면서 마주친 현실적인 고민들

처음엔 AI 에이전트마다 신원을 발급하고 권한을 세분화하는 게 번거롭고 관리 포인트가 늘어나는 것처럼 느껴졌습니다. 하지만 실제로 겪어보면, 이게 없으면 누가 어떤 AI 에이전트인지 구분 자체가 안 돼서 사고 발생 시 대응이 거의 불가능해요.

또 한 가지는 인증서 갱신과 폐기 정책을 어떻게 운영할지였습니다. 너무 자주 갱신하면 관리 부담이 커지고, 너무 느슨하면 보안 구멍이 되니까요. 우리 팀은 24시간 만료, 필요 시 즉시 폐기 정책을 적용했는데, 이 정도가 현실적인 타협점 같았습니다.

그리고 AI 에이전트가 생성하는 코드는 결국 사람이 검증해야 한다는 점도 잊으면 안 됩니다. AI가 반복적이고 단순한 작업을 자동화하는 데는 탁월하지만, 복잡한 아키텍처 설계나 보안 정책 결정은 여전히 사람의 판단이 필수적이니까요. [TypeScript Handbook](https://www.typescriptlang.org/docs/handbook/intro.html)에서도 AI 코딩 에이전트의 생산성 효과가 반복 작업에 집중된다고 명시하고 있습니다.

## 마무리하며: AI 에이전트 보안, 이제는 체계적으로 관리할 때

Teleport Agentic Identity Framework는 AI 에이전트가 기업 인프라에 안전하게 접근하고 활동할 수 있도록 신원과 권한을 명확히 분리해 줍니다. 실제로 도입해보면 초기 설정과 운영에 신경 쓸 부분이 많지만, 사고 발생 시 추적과 대응이 훨씬 수월해지는 걸 체감할 수 있어요.

그리고 AI가 생성한 코드는 반드시 별도의 보안 검증 프로세스를 거쳐야 하며, 권한 관리와 인증서 정책은 최소 권한 원칙과 주기적 갱신·폐기 정책을 준수하는 게 필수입니다.

우리 팀 경험을 바탕으로, AI 에이전트 보안에 관심 있는 동료 엔지니어분들이 Teleport Agentic Identity Framework를 실무에 적용할 때 도움이 되길 바랍니다.
