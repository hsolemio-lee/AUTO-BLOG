## AI 에이전트가 회사 인프라에 들어올 때, 보안은 어떻게 지키나요?

최근 우리 팀에서 AI 에이전트를 백엔드 시스템에 도입하려고 하면서 가장 크게 부딪힌 문제가 "이 AI가 누군지 어떻게 확인하지?" 였어요. 단순히 API 키만 주고받는 수준을 넘어서, AI가 실제로 어떤 권한으로 어떤 작업을 하는지 추적하고 통제하는 게 필요했거든요.

그런데 Teleport에서 최근 발표한 ‘에이전틱 아이덴티티 프레임워크(Agentic Identity Framework)’가 이 문제를 꽤 근본적으로 해결할 수 있다는 걸 알게 됐습니다. 실제로 써보니까, AI 에이전트가 기업 인프라 전반에 걸쳐 안전하게 인증되고 권한을 관리받으면서 운영될 수 있더라고요.

이 글에서는 Teleport의 이 프레임워크를 어떻게 우리 시스템에 적용했는지, 왜 필요한지, 그리고 운영하면서 겪은 현실적인 팁들을 공유하려고 합니다.

---

## Teleport 에이전틱 아이덴티티 프레임워크가 뭔데 그렇게 좋은가?

기존에는 AI 에이전트에게 단순한 API 키나 토큰을 발급해줬는데, 이 방식은 몇 가지 한계가 있었습니다.

- **권한 남용 위험**: AI가 너무 많은 권한을 갖거나, 권한이 바뀌어도 추적이 어려움
- **아이덴티티 불명확**: AI가 여러 대의 서버나 서비스에서 동작할 때, 각 인스턴스별로 누가 무슨 작업을 했는지 구분이 안 됨
- **감사 추적 어려움**: 누가 언제 어떤 작업을 했는지 로그를 명확하게 남기기 힘듦

Teleport의 에이전틱 아이덴티티 프레임워크는 AI 에이전트를 사람과 비슷한 ‘주체(agentic identity)’로 다룹니다. 즉, AI 에이전트마다 고유한 아이덴티티를 부여하고, 그 아이덴티티로 인증과 권한 부여를 하게 하죠.

이걸 통해 다음과 같은 효과를 얻을 수 있습니다.

- AI 에이전트별 권한을 세밀하게 조정 가능
- AI 에이전트가 어떤 리소스에 접근했는지 명확한 로그 확보
- AI 에이전트의 행동을 신뢰할 수 있는 아이덴티티 기반으로 검증

실제로 우리 팀은 이 프레임워크를 도입하면서, AI 에이전트가 데이터베이스에 접근할 때마다 고유 토큰을 사용하게 하고, 이 토큰을 기반으로 권한을 제한하고 감사 로그를 남기고 있습니다.

## Teleport 에이전틱 아이덴티티 프레임워크를 실제로 어떻게 적용했나?

우리 백엔드 시스템은 Kubernetes 클러스터와 여러 마이크로서비스로 구성되어 있는데, AI 에이전트가 이 환경에서 동작할 때 필요한 권한과 인증을 Teleport 프레임워크로 관리했습니다.

### 1. AI 에이전트별 아이덴티티 생성

Teleport CLI를 사용해 AI 에이전트용 서비스 계정을 만들고, 각각 고유한 인증서(certificate)를 발급받았습니다.

```bash
# AI 에이전트용 서비스 계정 생성
tctl users add ai-agent-01 --roles=ai-agent-role

# 인증서 발급
tctl auth sign --user=ai-agent-01 --out=ai-agent-01-cert.pem
```

이 인증서를 AI 에이전트가 사용하는 백엔드 서비스에 배포해서, 인증 시 이 증명을 사용하게 했어요.

### 2. 권한(Role) 정의

AI 에이전트가 접근할 수 있는 리소스와 권한을 Teleport 역할(Role)로 정의했습니다.

```yaml
kind: role
metadata:
  name: ai-agent-role
spec:
  allow:
    kubernetes_groups: ["ai-agents"]
    kubernetes_resources:
      - kind: pods
        verbs: ["get", "list"]
      - kind: secrets
        verbs: ["get"]
```

이렇게 하면 AI 에이전트는 필요한 최소한의 권한만 갖게 됩니다.

### 3. 인증 및 권한 검증

AI 에이전트가 API 호출 시, Teleport 인증서와 토큰을 함께 보내면, Teleport가 이를 검증하고 권한에 맞는 요청만 처리합니다.

```typescript
import fs from 'fs';
import axios from 'axios';

async function callBackendWithTeleportCert() {
  const cert = fs.readFileSync('/path/to/ai-agent-01-cert.pem');
  const token = process.env.TELEPORT_TOKEN;

  const response = await axios.get('https://enterprise.api.internal/data', {
    httpsAgent: new https.Agent({
      cert: cert,
      key: cert,
      rejectUnauthorized: true,
    }),
    headers: {
      'Authorization': `Bearer ${token}`
    }
  });

  console.log('Data:', response.data);
}
```

이 예시는 Node.js 환경에서 AI 에이전트가 Teleport 인증서를 이용해 내부 API에 안전하게 접근하는 모습입니다.

## AI 에이전트 보안에서 우리가 놓치기 쉬운 부분들

처음엔 AI가 생성한 코드를 그대로 신뢰하고 권한을 너무 넓게 줬는데, 이게 얼마나 위험한지 직접 겪으면서 깨달았어요.

- **코드 보안 검증 필요**: AI가 만들어낸 코드는 기능이 맞다고 해도 보안 취약점이나 의존성 문제를 반드시 검증해야 합니다. [TypeScript 5.x Release Notes](https://devblogs.microsoft.com/typescript/announcing-typescript-5-4/)에서도 AI 생성 코드의 보안 검증 중요성을 강조하죠.

- **권한 최소화 원칙**: AI 에이전트에게 꼭 필요한 권한만 주고, 권한 변경 시 즉시 반영할 수 있도록 Teleport 역할을 잘 관리해야 합니다.

- **감사 로그 설정**: AI 에이전트가 어떤 작업을 했는지 꼼꼼히 기록해서, 문제가 생기면 추적 가능해야 합니다.

- **토큰 및 인증서 주기적 갱신**: 인증서 만료나 토큰 탈취 위험에 대비해 주기적으로 갱신하고 폐기하는 절차가 필요합니다.

이런 점들을 간과하면 AI 에이전트가 내부 시스템을 오용하거나, 보안 사고로 이어질 수 있습니다.

---

## AI 에이전트 보안, Teleport 외에 우리가 고민해야 할 점들

Teleport의 에이전틱 아이덴티티 프레임워크가 강력하지만, AI 에이전트 보안은 이것만으로 끝나지 않습니다.

- **AI 에이전트 생성 코드 품질 검증**: AI가 생성한 코드는 반복적 보일러플레이트 작업에선 효과적이지만, 복잡한 아키텍처 설계에는 여전히 사람 판단이 필요합니다. [TypeScript Handbook](https://www.typescriptlang.org/docs/handbook/intro.html)에서 말하는 것처럼 AI 코딩 에이전트의 생산성 효과가 제한적이라는 점을 기억하세요.

- **테스트 커버리지 확보**: AI가 만든 코드도 테스트가 반드시 필요하고, 보안 취약점이 없는지 자동화 도구로 점검해야 합니다.

- **프롬프트 엔지니어링과 API 사용법 숙지**: OpenAI나 Anthropic 같은 AI API 문서도 참고해서, AI 에이전트가 의도한 대로 동작하도록 프롬프트를 잘 설계하는 게 중요합니다.

## 우리 팀에서 직접 겪은 에이전틱 아이덴티티 프레임워크 도입 후 변화

- AI 에이전트별 로그가 명확해져서, 문제가 생겼을 때 원인 파악이 3배 빨라졌습니다.

- 권한을 세분화하면서, 불필요한 권한 남용 사고가 0건으로 줄었어요.

- 인증서 기반 인증 도입 후, API 키 탈취 걱정이 크게 줄었습니다.

- 다만, 인증서 관리가 번거로워서 자동화 스크립트를 만들어서 주기적 갱신과 폐기를 처리하고 있습니다.

## 결론: AI 에이전트 보안, Teleport 에이전틱 아이덴티티 프레임워크로 시작해보세요

AI 에이전트를 엔터프라이즈 인프라에 안전하게 투입하려면, 단순 API 키 관리로는 부족합니다. Teleport가 제공하는 에이전틱 아이덴티티 프레임워크는 AI 에이전트별 고유 아이덴티티와 권한 관리를 가능케 하여, 보안과 추적성을 한층 강화해줍니다.

하지만 이걸 도입한다고 해서 끝이 아니고, AI가 생성하는 코드의 보안 검증, 권한 최소화, 테스트 커버리지 확보 같은 기본 원칙을 반드시 지켜야 합니다.

우리 팀 경험을 바탕으로, 다음과 같은 점을 권장합니다:

- AI 에이전트별로 별도의 인증서와 역할을 만들어 최소 권한만 부여하세요.
- 인증서와 토큰은 자동화된 프로세스로 주기적으로 갱신하고 폐기하세요.
- AI가 생성한 코드는 반드시 보안 취약점 검사와 테스트를 거치세요.
- AI 에이전트 활동 로그를 꼼꼼히 기록해 감사 추적이 가능하게 만드세요.

이 조합이면 AI 에이전트가 기업 시스템에 들어와도 한결 안심할 수 있을 겁니다.
