### AI 에이전트가 내 인프라에 들어온다면, 나는 어떻게 신원을 확인할까?

최근 AI 기반 코딩 에이전트들이 반복적이고 보일러플레이트한 작업을 훨씬 빠르게 처리하면서 개발 생산성을 끌어올리고 있죠. 하지만 이런 에이전트들이 기업 인프라 내부에서 자유롭게 움직인다면, 누가, 어떤 권한으로, 무엇을 하는지 명확히 관리하지 않으면 보안 사고로 직결될 수 있습니다. 그래서 Teleport가 발표한 Agentic Identity Framework가 꽤 흥미로운 대안으로 떠오르고 있는데요. 이번 글에서는 이 프레임워크가 왜 필요한지, 그리고 백엔드 엔지니어 입장에서 어떻게 도입할 수 있는지 실무 관점에서 풀어보겠습니다.

---

## Teleport Agentic Identity Framework, 대체 뭐가 다를까?

기존에도 AI 에이전트가 기업 시스템에 접근할 때 토큰이나 API 키 같은 인증 수단을 썼지만, 이 방법들은 에이전트가 실제 사람처럼 '신원'을 갖는다는 개념이 부족했어요. 그래서 권한 남용이나 추적 불가능한 작업이 발생하기 쉬웠죠.

Teleport의 Agentic Identity Framework는 AI 에이전트를 하나의 독립된 ‘주체(agentic identity)’로 인식합니다. 즉, 에이전트마다 고유한 신원과 권한을 부여하고, 그 신원으로 시스템에 접근하도록 강제하는 거죠. 이렇게 하면 누가 어떤 에이전트인지, 어떤 작업을 했는지 로그에 남기고, 권한 위임이나 회수가 훨씬 명확해집니다.

실제로 Teleport는 이 프레임워크를 통해 에이전트별로 세밀한 접근 제어와 감사(audit) 기능을 제공하며, 엔터프라이즈 환경에서 보안과 컴플라이언스를 동시에 만족시키는 게 가능하다고 밝혔습니다.

---

## AI 에이전트 신원 관리, 백엔드에서 어떻게 시작할까?

우선 Teleport를 인프라에 설치하고, AI 에이전트용 identity provider(IDP)를 설정하는 게 첫걸음입니다. Teleport는 오픈소스 기반으로, Kubernetes 클러스터나 VM, 데이터베이스 등 다양한 리소스에 통합할 수 있어요.

```yaml
# teleport.yaml 예시: AI 에이전트용 role 정의
kind: role
metadata:
  name: ai-agent
spec:
  allow:
    logins: ["agent-user"]
    node_labels:
      "agent": "true"
    rules:
      - resources: ["session"]
        verbs: ["read", "list"]
```

위처럼 AI 에이전트 전용 역할(role)을 정의하고, 이 역할에 맞는 권한만 부여합니다. 중요한 건 에이전트가 사람과 동일한 인증 메커니즘(예: OIDC, SAML)을 통해 인증받도록 설정하는 것인데요, 이렇게 하면 에이전트의 활동 내역이 사람과 동일한 수준으로 추적 가능합니다.

또한, Teleport는 에이전트가 사용하는 API 토큰이나 키를 주기적으로 회전(rotating)할 수 있게 지원해, 장기간 사용으로 인한 키 노출 위험도 줄입니다.

## AI가 만든 코드, 보안 검증은 어떻게 해야 할까?

AI가 코드를 생성하면 바로 배포하고 싶지만, 이때 보안 취약점 검증은 필수입니다. AI가 만든 코드가 동작한다고 해서 안전한 건 아니거든요.[TypeScript 5.x Release Notes](https://devblogs.microsoft.com/typescript/announcing-typescript-5-4/)에서도 강조하듯, 의존성 관리, 테스트 커버리지, 보안 취약점 스캔을 반드시 병행해야 합니다.

Teleport Agentic Identity Framework는 이런 점에서 AI 에이전트의 접근 권한을 최소화하고, 코드 배포나 수정 권한을 엄격히 제한할 수 있게 해줍니다. 예를 들어, AI 에이전트가 특정 환경에만 접근하도록 제한하고, 그 환경에서만 코드를 실행하도록 설정하는 거죠.

이런 방식은 AI가 만든 코드가 실수로라도 민감한 데이터베이스나 프로덕션 서버를 건드리지 못하게 막는 안전장치 역할을 합니다.

## 실무에서 Teleport Agentic Identity Framework 적용하며 겪은 한 가지 에피소드

우리 팀도 최근에 AI 코드 생성 도구를 도입하면서, 에이전트 신원 관리를 고민했는데요. 처음엔 단순히 API 키를 공유하는 수준이었어요. 그런데 한 번은 AI 에이전트가 테스트 환경의 설정을 실수로 바꿔서, 전체 테스트 파이프라인이 멈추는 일이 발생했습니다.

이후 Teleport를 도입해 에이전트별 역할과 권한을 명확히 나누고, 작업 로그를 실시간으로 모니터링하는 체계를 만들었죠. 덕분에 문제 발생 시 원인 추적이 훨씬 빨라졌고, 권한 남용 위험도 크게 줄었습니다.

처음엔 설정이 복잡해 보였지만, 실제로 겪어보니 AI 에이전트가 사람과 같은 신원으로 관리되는 게 얼마나 중요한지 뼈저리게 느꼈습니다.

## 실습: Teleport로 AI 에이전트용 Role과 Session 제한하기

아래는 Teleport CLI를 통해 AI 에이전트 역할을 만들고, 특정 노드에만 접근하도록 제한하는 간단한 예시입니다.

```bash
# ai-agent 역할 생성
cat <<EOF | tctl create -
kind: role
metadata:
  name: ai-agent
spec:
  allow:
    logins: ["agent-user"]
    node_labels:
      "env": "test"
    rules:
      - resources: ["session"]
        verbs: ["create", "read", "list"]
EOF

# ai-agent로 로그인 시도
tsh login --proxy=teleport.example.com --auth=oidc --user=ai-agent
```

이렇게 하면 ai-agent는 `env=test` 라벨이 붙은 노드에만 접근할 수 있고, 세션 생성과 조회 권한만 갖게 됩니다. 실제 환경에선 이 역할을 AI 에이전트의 IDP와 연동해 자동화하는 게 일반적입니다.
